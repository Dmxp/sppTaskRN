# Тестовое задание модуль управления СПП

Прототип веб-приложения для управления иерархическим справочником СПП, историчностью структуры, привязкой отделов, каскадным распределением сумм, сохранением вариантов и экспортом в Excel.

## Стек

- Backend: Python 3.13, FastAPI, SQLAlchemy 2, asyncpg
- Frontend: Next.js 15, React 19, TypeScript
- Database: PostgreSQL 17
- Cache: Redis 8
- Export: openpyxl
- Infrastructure: Docker Compose

## Запуск

docker compose up --build

После запуска:

- Frontend: http://localhost:3000
- Backend Swagger: http://localhost:8000/docs
- PostgreSQL: localhost:5432
- Redis: localhost:6379

## Сценарий проверки

1. Открыть http://localhost:3000.
2. Выбрать дату среза СПП.
3. Отметить один или несколько элементов дерева.
4. Ввести сумму и нажать `Выполнить`.
5. Проверить, что сумма распределилась по выбранным веткам и листьям.
6. Нажать `Сохранить`.
7. Убедиться, что список сохраненных расчетов обновился без перезагрузки страницы.
8. Открыть сохраненный расчет или скачать Excel.

## Redis → БД → Real-time

1. Пользователь выбирает элементы СПП и вводит сумму.
2. Backend получает актуальное дерево на выбранную дату.
3. Алгоритм распределяет сумму по выбранным веткам.
4. Результат сохраняется в Redis по ключу `calc:{uuid}` с TTL 30 минут.
5. После нажатия `Сохранить` backend переносит JSON-снимок из Redis в PostgreSQL.
6. После успешного сохранения backend отправляет WebSocket-событие всем подключениям с тем же `session_id`.
7. Frontend получает событие и обновляет список сохраненных вариантов без перезагрузки.

## Модель данных

### `spp_items`

Иерархический справочник СПП:

- `id`
- `parent_id`
- `code`
- `name`
- `status`
- `valid_from`
- `valid_to`

Историчность реализована через `valid_from` / `valid_to`. Для получения среза используется условие:

valid_from <= :target_date AND (valid_to IS NULL OR valid_to > :target_date)

### `departments`

Справочник отделов.

### `spp_department_links`

Связь отделов с элементами СПП. Связь также имеет период действия.

### `calculation_results`

Таблица сохраненных вариантов:

- `id`
- `session_id`
- `created_at`
- `status`
- `spp_version_date`
- `snapshot JSONB`

В `snapshot` хранится полный снимок распределенной структуры.

## Алгоритм распределения

Система принимает:

- массив ID выбранных элементов СПП;
- общую сумму.

Логика:

1. Общая сумма делится поровну между выбранными ветками.
2. Каждая выбранная ветка рекурсивно распределяет свою долю между дочерними элементами.
3. Листовые узлы получают конечные суммы.
4. Родительские узлы агрегируют суммы потомков.
5. Значения округляются до двух знаков.
6. Для компенсации копеечных расхождений последний элемент получает остаток.

## API

- `GET /api/dates` — список доступных дат изменений.
- `GET /api/tree?version_date=2025-06-01` — дерево СПП на дату.
- `POST /api/calculate` — выполнить расчет и сохранить результат в Redis.
- `POST /api/save/{redis_id}` — перенести расчет из Redis в PostgreSQL.
- `GET /api/calculations?session_id=...` — список сохраненных расчетов.
- `GET /api/calculations/{id}` — загрузить сохраненный расчет.
- `GET /api/export/{id}` — скачать Excel-файл.
- `WS /ws/{session_id}` — real-time уведомления.

## AI-workflow и инженерный контроль

При выполнении задания использовались ИИ-ассистенты как инструмент ускорения разработки и проверки архитектурных решений.

### Использованные инструменты

- ChatGPT — декомпозиция требований, проектирование структуры БД, генерация первичной версии кода.
- Ручная проверка — анализ соответствия требованиям тестового задания, проверка связности API и фронтенда.

### Примеры рабочих промтов

Напиши алгоритм рекурсивного распределения суммы по выбранным веткам дерева: сумма делится между выбранными узлами, затем внутри каждой ветки распределяется между листьями, родители агрегируют суммы потомков.

Сделай FastAPI endpoint, который сохраняет результат расчета в Redis с TTL, а затем отдельным endpoint переносит его в PostgreSQL и отправляет WebSocket-уведомление по session_id.

### Как проверялся и дорабатывался код

- Проверена полнота покрытия требований: PostgreSQL, Redis, Docker, Next.js, FastAPI, WebSocket, Excel.
- Схема БД проверена на разделение справочников и связей.
- Алгоритм вынесен в отдельный сервис, чтобы его можно было тестировать независимо от API.
- Redis используется только как временное хранилище, PostgreSQL — как постоянное.
- JSONB-снимок хранит всю структуру расчета, поэтому сохраненный вариант можно восстановить даже при изменении справочника СПП в будущем.

## Ограничения прототипа

- CRUD СПП не реализован, так как в задании он не требуется.
- Авторизация не реализована.
- Для демонстрации механизма WebSocket-синхронизации используется фиксированный идентификатор сессии `demo-session`.
- В реальном приложении `session_id` должен генерироваться для каждого пользователя отдельно и храниться в браузере либо передаваться через систему авторизации.
- Демо-данные загружаются через `postgres/init.sql` при первом старте контейнера PostgreSQL.
